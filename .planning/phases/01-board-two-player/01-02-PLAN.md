<!-- API contract — every fetch call must match these exactly -->

POST /games
  Request: (no body)
  Response 201: GameResponse

GET /games/{game_id}
  Response 200: GameResponse

POST /games/{game_id}/moves
  Request: { "position": 0-8, "player": "X"|"O" }
  Response 200: GameResponse
  Response 400: { "error": "cell_taken"|"wrong_turn"|"game_over"|"invalid_position", "detail": "..." }

POST /games/{game_id}/reset
  Response 200: GameResponse

GameResponse shape:
```json
{
  "id": "uuid-string",
  "board": ["X", "", "O", "", "", "", "", "", ""],
  "current_player": "X",
  "status": "in_progress",
  "winner": null,
  "winning_line": null,
  "move_count": 1
}
```

- board: array of 9 strings, each "X", "O", or "" (empty)
- status: "in_progress" | "x_wins" | "o_wins" | "draw"
- winner: "X" | "O" | null
- winning_line: array of 3 ints (e.g. [0,1,2]) | null
- current_player: "X" | "O" (whose turn it is next)

Board index layout:
  0 | 1 | 2
  ---------
  3 | 4 | 5
  ---------
  6 | 7 | 8

## CSS Requirements

The `<style>` block must include:

1. **Board layout**: `#board` uses `display: grid; grid-template-columns: repeat(3, 1fr);`
   with a fixed width (e.g., 300px) and `gap: 4px;`.
2. **Cell styling**: `.cell` is square (100px × 100px or use aspect-ratio), has a visible
   border, cursor pointer, centers text with large font (e.g., 2.5rem), and has a
   background color.
3. **Winning highlight**: `.cell.winning` has a distinct background color (e.g.,
   `#90EE90` light green or `#FFD700` gold). This class is applied to winning cells.
4. **X and O colors**: `.cell.player-X` has one color (e.g., `#2196F3` blue),
   `.cell.player-O` has another (e.g., `#F44336` red). Apply via text color.
5. **Error message**: `.error` has red text color.
6. **Outcome message**: `#outcome-message` has larger/bold text and distinct color
   (e.g., green for win, gray for draw).
7. **Disabled cell appearance**: `.cell.disabled` has `cursor: not-allowed; opacity: 0.7;`
   (applied when game is over).
8. **General layout**: Center the `#app` container. Add spacing between elements.

## JavaScript Requirements

The `<script>` block must implement:

### State Variables
```javascript
let gameId = null;   // Current game UUID from API
let gameState = null; // Latest GameResponse from API
```

### API Helper Function
```javascript
async function apiCall(url, method = 'GET', body = null) {
  const opts = { method, headers: { 'Content-Type': 'application/json' } };
  if (body) opts.body = JSON.stringify(body);
  const res = await fetch(url, opts);
  const data = await res.json();
  if (!res.ok) throw data;  // { error, detail }
  return data;
}
```

### Core Functions

1. **`createGame()`** — calls `POST /games`, stores `gameId` and `gameState`, calls
   `renderBoard()`. Called on page load and when "New Game" is clicked.

2. **`makeMove(position)`** — calls `POST /games/${gameId}/moves` with
   `{ position, player: gameState.current_player }`. On success: update `gameState`,
   call `renderBoard()`, clear error. On 400 error: display `data.detail` in
   `#error-message` (PLAY-02). Guard: if `gameState.status !== 'in_progress'`, return
   early (don't call API).

3. **`resetGame()`** — calls `POST /games/${gameId}/reset`, updates `gameState`,
   calls `renderBoard()`, clears error and outcome messages.

4. **`renderBoard()`** — clears `#board`, creates 9 `.cell` divs. For each cell at
   index `i`:
   - Set `textContent` to `gameState.board[i]` (will be "X", "O", or "")
   - Set `data-index` attribute to `i`
   - If cell is non-empty, add class `player-X` or `player-O` accordingly
   - Add click handler that calls `makeMove(i)`
   - If `gameState.winning_line` is not null and includes `i`, add class `winning` (UI-04)
   - If `gameState.status !== 'in_progress'`, add class `disabled`

   After rendering cells, update:
   - `#turn-display` text to `gameState.current_player`
   - If `gameState.status === 'x_wins'`: show `#outcome-message` with text "X wins!"
   - If `gameState.status === 'o_wins'`: show `#outcome-message` with text "O wins!"
   - If `gameState.status === 'draw'`: show `#outcome-message` with text "It's a draw!"
   - If `gameState.status === 'in_progress'`: hide `#outcome-message`

### Event Wiring
```javascript
document.getElementById('new-game-btn').addEventListener('click', resetGame);
// On page load:
createGame();
```

### Error Display Logic
- When `makeMove` catches a 400 error, set `#error-message` textContent to the
  error's `detail` field and set `display: block`.
- On successful move or new game, hide `#error-message` (`display: none`).

### Mode Selector Behavior (MODE-01, MODE-02)
- The "Two Player" radio is checked by default.
- The "vs Computer" radio is disabled with text "(coming soon)".
- No JS handler needed for mode yet — two-player is the only functional mode.
- The mode selector must be present and visible in the DOM for MODE-01.

## Key constraints
- All game state comes from API responses. Do NOT duplicate win detection, turn logic,
  or move validation in JavaScript. The JS only reads `gameState` from API responses
  and renders accordingly (INT-02).
- Use `fetch()` with relative URLs (e.g., `/games`, not `http://localhost:8000/games`)
  since the page is served from the same origin (INT-03).
- The `player` field sent to `POST /games/{id}/moves` must be `gameState.current_player`
  — the frontend just passes along whatever the API says the current player is.
